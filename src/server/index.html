How Railgun Works
TL;DR: You define your AI agents and their steps in a simple YAML file. Railgun runs them in order, one by one, passing each agent's output to the next. No agent decides the flow. The file does. That's why it's reliable.

The Simple Version

  • Think of it as an assembly line for AI agents
  • Each agent gets a fresh start |no bloated context windows
  • Work passes from one agent to the next via a "clipboard"
  • If step 4 breaks, you fix step 4 |not the whole conversation
  • The board above shows your workflow steps as columns, with runs as cards

How a Run Works

  • A workflow defines steps in sequence (the columns you see above)
  • Each step is assigned to a specific agent with its own role
  • Agents poll for work using a two-phase system |cheap check first, expensive model only when there's work
  • When an agent completes its step, the output becomes input for the next step
  • Everything is tracked in SQLite |every step, every output, every failure

Why This Matters

  • Cost control |each agent only uses tokens for its one job, not re-reading entire histories
  • Reliability |deterministic YAML sequence, not AI deciding what to do next
  • Debuggable |click any card to see exactly what the agent received and produced
  • Retry logic |failed steps retry automatically with configurable limits
  • Health monitoring |the Health Monitor detects stuck, stalled, or dead runs

Key Commands

  • clawsecure-railgun workflow list |see available workflows
  • clawsecure-railgun workflow run <name> "task" |start a run
  • clawsecure-railgun workflow status <query> |check run status
  • clawsecure-railgun dashboard start |launch this dashboard
  • clawsecure-railgun medic run |run health checks manually
